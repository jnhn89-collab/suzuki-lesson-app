# 분기/학기 성과보고서 웹앱 구현 계획서

## 1. 목표

선생님이 웹으로 접속해 학생별 분기/학기 평가를 입력하고, 학부모에게 공유 가능한 모바일 보고서 링크를 생성한다. 학부모는 공유 URL에 접속한 뒤 학생 확인 정보와 비밀번호를 입력하고, 해당 학생의 정보와 선생님 피드백 보고서를 확인한다. 보고서는 선생님 화면과 학부모 화면 모두에서 모바일 열람과 인쇄용 PDF 다운로드가 가능해야 한다.

## 2. 최종 사용자 시나리오

### 2.1 선생님

1. 선생님이 웹앱에 로그인한다.
2. 학생 목록에서 학생을 선택한다.
3. 평가 기간을 선택한다.
   - 이번 학기
   - 이번 분기
   - 1분기/2분기/3분기/4분기
   - 기간 직접 입력
4. 영역별 점수, 기간 내 레슨 수, 정리한 곡, 현재 진도, 피드백을 입력한다.
5. 프리셋 문장을 활용해 빠르게 보고서를 작성한다.
6. 미리보기에서 모바일/인쇄 레이아웃을 확인한다.
7. 보고서를 저장하고 학부모 공유 링크를 생성한다.
8. 필요하면 PDF를 다운로드한다.
9. 학부모에게 공유 링크와 안내 문구를 보낸다.

### 2.2 학부모

1. 학부모가 공유 URL에 접속한다.
2. 인증 페이지에서 다음 정보를 입력한다.
   - 학생 생년월일 6자리
   - 학부모 휴대폰 뒷자리 4자리
   - 보고서 비밀번호 또는 PIN
3. 인증 성공 시 해당 학생의 기본 정보와 성과보고서를 본다.
4. 모바일에서 편하게 읽거나 PDF로 다운로드한다.
5. 인증 실패 시 제한된 횟수만 재시도할 수 있다.

## 3. 권장 기술 구조

현재 Google Apps Script 앱은 유지하되, 성과보고서 공유 기능은 별도 웹앱으로 분리한다. 이유는 공유 링크, 학부모 인증, DB 보안, PDF 생성, 배포 관리가 필요하기 때문이다.

### 3.1 추천 스택

- 코드 저장소: GitHub
- 프론트엔드/서버 배포: Vercel
- 앱 프레임워크: Next.js
- DB/Auth/Storage: Supabase
- PDF 생성:
  - 1차: 브라우저 `window.print()` 기반 PDF 저장
  - 2차: 서버 Playwright 기반 PDF 렌더링

### 3.2 URL 구조

```text
/teacher
  선생님 대시보드

/teacher/students
  학생 목록 및 관리

/teacher/reports/new
  새 보고서 작성

/teacher/reports/:reportId
  보고서 수정/미리보기/PDF 다운로드/공유 링크 관리

/r/:publicToken
  학부모 인증 페이지

/r/:publicToken/view
  인증 성공 후 학부모 보고서 열람
```

## 4. 데이터 설계

### 4.1 teachers

선생님 계정 정보.

```text
id
email
name
studio_name
created_at
updated_at
```

### 4.2 students

학생 기본 정보. 생년월일은 인증에 쓰이므로 평문 노출을 최소화한다.

```text
id
teacher_id
name
birth_ymd
birth_yyMMdd_hash
age_group
current_book
current_piece
status
created_at
updated_at
```

초기 버전에서는 선생님 화면에서 생년월일을 볼 수 있어야 하므로 `birth_ymd`를 저장할 수 있다. 다만 학부모 인증 비교에는 `birth_yyMMdd_hash`를 사용한다.

### 4.3 parents

학부모 인증 정보.

```text
id
student_id
parent_name
phone_last4_hash
phone_last4_hint
relation
status
created_at
updated_at
```

`phone_last4_hint`는 화면에 표시할 수 있는 힌트가 필요할 때만 사용한다. 예: `010-****-1234`처럼 전체 번호를 저장하지 않는 방식.

### 4.4 reports

성과보고서 본문과 상태.

```text
id
teacher_id
student_id
period_name
period_start
period_end
total_lessons
completed_pieces
current_piece
scores_json
focus_tags_json
strengths
growth_area
home_support
practice_plan
daily_minutes
daily_reps
teacher_summary
status
created_at
updated_at
published_at
expires_at
```

`status` 예:

```text
draft
published
archived
revoked
```

### 4.5 report_links

공유 링크와 인증 설정.

```text
id
report_id
public_token
password_hash
max_attempts
failed_attempts
locked_until
expires_at
created_at
last_accessed_at
revoked_at
```

`public_token`은 학생 ID나 보고서 ID를 노출하지 않는 충분히 긴 랜덤 문자열을 쓴다.

### 4.6 report_access_logs

접속 이력.

```text
id
report_id
public_token
success
failure_reason
ip_hash
user_agent
created_at
```

## 5. 보안 설계

### 5.1 학부모 링크 보안

공유 URL만으로 보고서가 열리면 안 된다.

```text
/r/:publicToken
```

접속 후 다음 값을 확인한다.

```text
학생 생년월일 6자리
학부모 휴대폰 뒷자리 4자리
보고서 비밀번호 또는 PIN
```

비교는 해시값 기준으로 한다.

### 5.2 실패 제한

- 실패 5회 시 30분 잠금
- 만료된 링크는 열람 불가
- 선생님이 공유 링크를 폐기할 수 있어야 함
- 인증 성공 후에도 짧은 세션 토큰으로만 보고서 view 접근

### 5.3 권한

- 선생님은 본인 학생과 본인 보고서만 접근 가능
- 학부모는 인증한 `publicToken`에 해당하는 보고서만 접근 가능
- Supabase RLS 또는 서버 API에서 teacher ownership 체크 필수

### 5.4 민감정보 저장 원칙

- 학부모 휴대폰 전체 번호는 저장하지 않는다.
- 휴대폰 뒷자리 4자리는 해시 저장한다.
- 생년월일 인증값도 해시 저장한다.
- 보고서 본문은 학생 개인정보이므로 공개 URL에서 직접 조회되지 않게 한다.

## 6. 화면 설계

### 6.1 선생님 대시보드

기능:

- 학생 목록
- 최근 작성 보고서
- 초안 보고서
- 발행된 보고서
- 만료/폐기된 링크 상태 확인

### 6.2 학생 관리

기능:

- 학생 등록
- 학생 이름, 생년월일, 현재 진도, 학부모 정보 입력
- 학부모 휴대폰 뒷자리 등록
- 학생 비활성화

### 6.3 보고서 작성 화면

현재 `report-prototype/index.html`의 입력 경험을 앱 화면으로 이전한다.

필수 입력:

- 학생
- 평가 기간
- 기간 시작일/종료일
- 현재 진도
- 기간 내 레슨 수
- 완료/정리한 곡 수
- 영역별 점수
- 이번 기간 키워드
- 좋아진 점
- 다음 성장 포인트
- 가정 도움말
- 다음 기간 연습 방향

편의 기능:

- 문장 프리셋
- 모바일 미리보기
- 인쇄 미리보기
- 자동 저장
- 초안 저장
- 발행 전 검증

### 6.4 보고서 미리보기

선생님 화면에서 다음 기능 제공:

- 모바일 미리보기
- 인쇄/PDF 미리보기
- PDF 다운로드
- 공유 링크 생성
- 공유 링크 복사
- 링크 폐기
- 링크 만료일 설정

### 6.5 학부모 인증 화면

모바일 우선 화면.

입력:

- 학생 생년월일 6자리
- 학부모 휴대폰 뒷자리 4자리
- 보고서 비밀번호/PIN

상태:

- 잘못된 입력
- 잠금 상태
- 만료된 링크
- 폐기된 링크
- 인증 성공

### 6.6 학부모 보고서 화면

입력 패널 없이 보고서만 표시한다.

기능:

- 학생 이름
- 평가 기간
- 현재 진도
- 기간 요약
- 영역별 기록
- 선생님 피드백
- 다음 기간 연습 방향
- PDF 다운로드

## 7. PDF 설계

### 7.1 1차 구현

브라우저 인쇄 기능을 사용한다.

- `@media print` CSS 정리
- 선생님 화면의 입력 패널 숨김
- 보고서 영역만 출력
- 모바일 화면에서도 PDF 저장 가능

장점:

- 구현 빠름
- 서버 비용 없음

한계:

- 기기/브라우저에 따라 PDF 품질 차이 있음

### 7.2 2차 구현

서버에서 HTML을 렌더링해 PDF 생성.

- Vercel 서버 함수 또는 별도 Node 서버
- Playwright/Chromium 사용
- 고정된 A4 레이아웃
- `reports/:id/pdf` 다운로드 API 제공

장점:

- 일관된 PDF 품질
- 학부모/선생님 모두 같은 파일 다운로드

## 8. 개발 단계

### Phase 1. 정적 프로토타입 정리

목표: 현재 HTML 프로토타입을 실제 앱으로 옮기기 쉬운 형태로 정리한다.

작업:

- `report-prototype` 유지
- 디자인/문구 확정
- 모바일 학부모 보기 확정
- 프리셋 문장 추가 정리
- 보고서 데이터 구조 확정

완료 기준:

- 선생님 입력 화면이 기간 보고서 기준으로 동작
- 학부모 보기 모드가 모바일에서 자연스럽게 표시
- PDF 인쇄 스타일 확인

### Phase 2. Next.js 앱 생성

목표: 실제 웹앱 기반을 만든다.

작업:

- Next.js 프로젝트 구성
- 라우팅 구성
- 공통 레이아웃/스타일 정리
- 현재 보고서 프로토타입을 컴포넌트화
- GitHub/Vercel 배포 연결

완료 기준:

- 배포 URL 접속 가능
- `/teacher/reports/new`에서 보고서 작성 가능
- `/r/demo-token`에서 학부모용 화면 확인 가능

### Phase 3. Supabase DB 연결

목표: 학생, 학부모, 보고서를 DB에 저장한다.

작업:

- Supabase 프로젝트 생성
- 테이블 생성
- RLS 정책 또는 서버 API 권한 체크 설계
- 학생 CRUD
- 보고서 draft 저장
- 보고서 published 저장

완료 기준:

- 선생님이 학생을 등록할 수 있음
- 선생님이 보고서를 저장/수정할 수 있음
- 저장된 보고서가 새로고침 후에도 유지됨

### Phase 4. 선생님 인증

목표: 아무나 선생님 화면에 접근하지 못하게 한다.

작업:

- Supabase Auth 또는 이메일 OTP 로그인
- teacher profile 생성
- teacher_id 기준 데이터 필터링
- 로그인/로그아웃 UI

완료 기준:

- 로그인한 선생님만 대시보드 접근 가능
- 선생님은 본인 학생/보고서만 볼 수 있음

### Phase 5. 공유 링크와 학부모 인증

목표: 보고서별 보안 링크를 발급하고 학부모 인증을 구현한다.

작업:

- `public_token` 생성
- 보고서 비밀번호/PIN 해시 저장
- 생년월일 6자리 해시 비교
- 휴대폰 뒷자리 4자리 해시 비교
- 실패 횟수 제한
- 링크 만료/폐기 처리
- 인증 성공 세션 처리

완료 기준:

- 링크만으로 보고서가 열리지 않음
- 올바른 인증값 입력 시 보고서 열람 가능
- 잘못된 입력 반복 시 잠김
- 만료/폐기된 링크는 접근 불가

### Phase 6. PDF 다운로드

목표: 선생님과 학부모 모두 PDF를 받을 수 있게 한다.

작업:

- 1차: 브라우저 print CSS 완성
- 다운로드 버튼 추가
- 2차: 서버 PDF 생성 API 검토 및 구현

완료 기준:

- 선생님 보고서 화면에서 PDF 저장 가능
- 학부모 보고서 화면에서 PDF 저장 가능
- 모바일에서도 인쇄/PDF 버튼이 동작

### Phase 7. 운영 품질

목표: 실제 학부모에게 보낼 수 있는 수준으로 다듬는다.

작업:

- 모바일 QA
- 긴 이름/긴 문장 레이아웃 확인
- 인증 실패/만료/폐기 상태 UX 확인
- 접근 로그 확인
- 개인정보 노출 점검
- README 배포 문서 작성

완료 기준:

- 실제 배포 URL로 선생님 작성 가능
- 생성된 보고서 링크를 학부모에게 전달 가능
- 학부모 인증 후 모바일 열람 가능
- PDF 다운로드 가능

## 11. 리스크와 대응

### 개인정보 리스크

학생 정보와 학부모 인증 정보가 있으므로 공개 정적 웹만으로 처리하면 안 된다.

대응:

- DB 접근은 서버 API 또는 RLS로 제한
- 인증값 해시 저장
- 공유 링크에 학생 ID 노출 금지

### 링크 유출 리스크

공유 링크가 다른 사람에게 전달될 수 있다.

대응:

- 링크 접속 후 추가 인증 필수
- 링크 만료일
- 실패 제한
- 링크 폐기 기능

### PDF 품질 리스크

브라우저마다 PDF 결과가 다를 수 있다.

대응:

- 1차는 print CSS로 빠르게 구현
- 운영 안정화 후 서버 PDF 렌더링으로 개선

### 선생님 입력 부담

보고서 작성이 길어지면 실제 사용성이 떨어진다.

대응:

- 프리셋 문장 강화
- 자동 저장
- 필수 입력 최소화
- 이전 보고서 복사 기능을 후속으로 추가

## 12. 완료 정의

아래가 모두 가능하면 1차 완료로 본다.

- 선생님이 웹에 접속해 로그인할 수 있다.
- 선생님이 학생을 등록할 수 있다.
- 선생님이 분기/학기 성과보고서를 작성하고 저장할 수 있다.
- 선생님이 보고서를 PDF로 저장할 수 있다.
- 선생님이 학부모 공유 링크를 생성할 수 있다.
- 학부모가 공유 링크에 접속하면 인증 페이지가 먼저 나온다.
- 학부모가 학생 생년월일 6자리, 휴대폰 뒷자리 4자리, 비밀번호/PIN을 입력하면 보고서를 볼 수 있다.
- 학부모 화면은 모바일에서 읽기 좋다.
- 학부모도 PDF를 다운로드할 수 있다.
- 잘못된 인증 시 재시도 제한이 있다.
- 만료되거나 폐기된 링크는 열람할 수 없다.
